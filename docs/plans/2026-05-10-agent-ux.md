# rest-magpie agent-UX plan — 2026-05-10

> **Sequencing across both plans is in `2026-05-10-roadmap.md`. Read that first.** This doc holds agent-UX detail (what we build, why each suggestion was taken/deferred/rejected, spec amendments). The work itself is split across roadmap Phase 1 (v0.1.2 patch — DONE) and Phase 2 (v0.2.0 bundle — in progress). Don't act on this doc in isolation — go through the roadmap.

**Sources:**
1. Claude Code agent (no-context Elasticsearch exploration, reflexively `include_body: true` everywhere). 7 suggestions about description/body-mode UX.
2. GPT-5.4 agent (different real-world session, hit chunked-multipart compat + path-flow ambiguity). 9 suggestions about run-mode docs, debug tooling, file-payload alternatives.

The core insight from both: **the right path (`include_body: "auto"` → schema → `http_read` with jq, plus same-path bind mount in Docker) is already what the project supports — but agents don't read enough of the description, and the tooling doesn't expose enough runtime context for them to self-correct.** Two surfaces to fix: (a) tool descriptions and structured runtime introspection, (b) additive features that close the corner cases neither agent could route around.

## Release sequencing

### v0.1.2 — DONE (tagged 2026-05-10)

3 PRs shipped: #17 (tool descriptions), #18 (README run-modes), #19 (richer path errors). #20 was the version bump.

### v0.2.0 — agent-UX bundle (additive spec amendment + breaking rename)

**Theme:** "make the right path also the easiest path." Spec §3-4 amendments + several backward-compatible additions + ONE breaking rename (zero-install migration cost = free upgrade now).

PRs (sequential — touch overlapping files):

- `feat/body-mode-rename` — replace `include_body: true | false | "auto"` with `body_mode: "auto" | "schema" | "head" | "inline"`.
  - Spec §4 amendment: param renamed, enum redefined, default `"auto"`.
  - Migration mapping (CHANGELOG-only, no runtime alias):
    - `include_body: true` → `body_mode: "inline"`
    - `include_body: false` → `body_mode: "schema"`
    - `include_body: "auto"` → `body_mode: "auto"`
    - omitted → `body_mode: "auto"` (unchanged)
  - Validation: if request contains the legacy field name `include_body`, return `invalid_input` with message: "`include_body` is no longer supported. Use `body_mode: 'auto' | 'schema' | 'head' | 'inline'` (default 'auto'). See CHANGELOG for migration." Don't silently accept it — fail loud.
  - Tool description: explicitly explain mode names by what you GET (schema = structure only) and what HAPPENS (inline = body enters your context).
  - Tests: every existing test referencing `include_body` updated. Add legacy-field-rejection test.

- `feat/body-mode-head` — add `"head"` mode behavior.
  - Behavior: schema (as today) + a `body_preview` field on the response: arrays truncated to first N items (`MAGPIE_HEAD_PREVIEW_ITEMS`, default 5), strings truncated to first M chars (`MAGPIE_HEAD_PREVIEW_STRING`, default 200), object scalars passed through. Each truncated container gets a sibling marker (e.g. `_truncated: { kind: "array", original_length: 5000, included: 5 }`).
  - `auto` semantics extended: when 8 KB < body ≤ `MAGPIE_HEAD_PREVIEW_THRESHOLD` (default 64 KB), auto upgrades to `head` instead of pure schema-only. Above threshold → schema-only as today. Documented as a transparent middle tier.
  - Tests: golden snapshots for nested array-of-objects, deep object trees, mixed types.

- `feat/schema-next-step-hints` — auto-generate `next_step_hints` on the response.
  - Top-level shape detection: if `body` is an array → emit suggestions for `length`, head/tail slices, `map({k1,k2})`, `sort_by`. If object with one or more array-valued fields → suggestions on those fields. If plain object → key projection example.
  - Always advisory, not authoritative — README/spec wording: "rest-magpie suggests common jq patterns based on shape; the agent picks one or writes its own."
  - Implementation: pure function `inferHints(schema, top_level_shape) → string[]` in `src/core/schema/hints.ts`. ~5-7 hint templates max.
  - Hide hints when body is non-JSON (text/binary).

- `feat/inline-cap-with-friendly-error` — guardrail on `body_mode: "inline"` for big bodies.
  - New env: `MAGPIE_INLINE_BODY_CAP` (default 256 KB).
  - When `body_mode: "inline"` AND `body_bytes > cap` → return `body_too_large_for_inline` error envelope (new error kind in spec §6) with payload that includes `cache_id` (the body IS still cached) and a suggested mask string.
  - Test: response 300 KB + `body_mode: "inline"` → error envelope, but cache entry still exists.

- `feat/inclusion-metadata` — surface inclusion reason in `meta`.
  - Replace `meta.body_included: boolean` with `meta.body_inclusion: { resolved_mode: "schema" | "head" | "inline", inline_threshold_bytes: number, head_preview_threshold_bytes: number, reason?: string }`. Since we're already breaking on the rename, no alias kept — clean cut.
  - Reason populated only when the resolved mode wasn't the obvious one (e.g. "body 13558B exceeded inline threshold 8192B; auto downgraded to head" or "body 312KB exceeded inline cap 256KB").

- `feat/server-info-tool` (GPT-5.4 P0-3) — new 4th MCP tool `server_info`.
  - Spec §3 amendment: add `server_info` to the canonical tool list.
  - No params. Returns: `version`, `runtime` (`"npx"` / `"docker"` / `"unknown"`, detected via `/.dockerenv` + heuristics), `cwd`, `files_root` (or null), `effective_limits` (every `MAGPIE_*` env var resolved to its actual numeric/boolean value).
  - One-shot, idempotent, no side effects, no cache. Description leans on debug-utility framing: "Use this when path is rejected unexpectedly, or to confirm which container/host the server is actually running in."
  - Tests: matrix of env-var settings → assert effective_limits reflects them.

- `feat/multipart-content-base64` (GPT-5.4 P1-4) — alternative file payload mode.
  - Spec §4 amendment: `multipart.files.<name>` schema becomes a `oneOf`: `{ path }` OR `{ content_base64, filename?, content_type? }`. Exactly one required.
  - Implementation: when `content_base64` is present, decode into a `Buffer`, build the multipart frame from buffers (still streamed via `Readable.from` async generator, just yielding from `Buffer` instead of `createReadStream`).
  - Size guardrail: `MAGPIE_MAX_INLINE_FILE_BYTES` (default 10 MB pre-base64, ≈13.3 MB encoded). Refuse beyond cap with `invalid_input`.
  - `MAGPIE_FILES_ROOT` validation does NOT apply to `content_base64` (no path to canonicalize) — explicitly documented as a feature: this is the recommended mode for remote MCP scenarios.
  - Tests: inline upload, mixed `path` + `content_base64` files in same request, oversize rejection, `content_type` inference fallback to `application/octet-stream`.

Spec §3-4 amendments go in a single doctrine-PR before any feature PR opens. Each subsequent PR references the amendment commit.

Ship target: v0.2.0 tagged after all 7 PRs + amendment merge.

## Out of scope for v0.2 / explicit non-goals

- **Per-call `force: true` override on inline cap** — too niche, env var is enough.
- **Auto-execution of suggested jq** — would turn `next_step_hints` into magic. They stay advisory; the agent calls `http_read` itself.
- **Streaming `head` previews** — for v0.2 we still buffer-then-truncate. Streaming-truncate is a v0.3 conversation if it ever matters.
- **Non-chunked multipart compat mode** (GPT-5.4 P2-7) — defer to v0.3 backlog.
- **Artifact-oriented file flow** (GPT-5.4 P2-8) — **rejected outright**. Adds stateful artifact lifecycle (creation, TTL, cleanup, retention env vars, per-agent visibility) to solve remote-MCP file asymmetry. Cheaper alternative: `content_base64` on upload (in this v0.2 bundle) plus a future `download_mode: "inline_base64"` toggle on `http_request` (one-line addition when/if needed) closes the same gap with zero state.
- **Curl export / repro helper** (GPT-5.4 P2-9) — defer to v0.3 backlog.

## Spec amendments (single PR before v0.2 work starts)

Open `docs/specs/2026-05-09-rest-magpie-design.md` and amend:
- §3 — add `server_info` to the canonical tool list with brief description.
- §4 — **`include_body` parameter removed; replaced by `body_mode: "auto" | "schema" | "head" | "inline"` (default `"auto"`).** New result-shape field `body_preview` (when resolved mode is `"head"`). New `meta.body_inclusion: { resolved_mode, inline_threshold_bytes, head_preview_threshold_bytes, reason? }` shape replacing `meta.body_included: boolean`. `multipart.files.<name>` becomes `oneOf({path}, {content_base64, filename?, content_type?})`.
- §5 — schema-renderer output now optionally embeds `next_step_hints: string[]` (always present, possibly empty).
- §6 — new error kind `body_too_large_for_inline` with example payload. New error kind `unsupported_field` returned when legacy `include_body` is sent.
- §10 — new env vars: `MAGPIE_INLINE_BODY_CAP`, `MAGPIE_HEAD_PREVIEW_ITEMS`, `MAGPIE_HEAD_PREVIEW_STRING`, `MAGPIE_HEAD_PREVIEW_THRESHOLD`, `MAGPIE_MAX_INLINE_FILE_BYTES`. Removed: `MAGPIE_AUTO_INCLUDE_BODY_BYTES` is renamed to `MAGPIE_INLINE_THRESHOLD_BYTES` (the boundary below which `auto` mode resolves to `inline`). Document defaults.
- §13 — expand Docker section into full "Run modes & file locality" matrix (npx / Docker / remote MCP). Adds chunked-multipart compat note.
- §15 — move "v0.2 agent-UX" out of backlog into "Done in v0.2". Update v0.3 backlog with non-chunked multipart, artifact handles, curl export.

## Open questions

- For `head` mode, do we truncate strings AND arrays, or only arrays? Strings can carry payload too (HTML, base64). **Lean: both, with separate env knobs.**
- Should the hard inline cap also apply to `body_mode: "head"` (where preview itself could be large if we don't bound it)? **Lean: yes, always apply final-output cap regardless of mode — defense in depth.**

### Phase 2 — v0.2.0 bundle (tag `v0.2.0`) — IN PROGRESS

**Theme:** "make the right path also the easiest path." Spec §3-4-5-6-10-13 amendments + several backward-incompatible additions.

**Detail:** see `2026-05-10-agent-ux.md` § "v0.2.0 — agent-UX bundle".

**Phase 2.0 — spec amendment (must be first):**
- Single PR amends `docs/specs/2026-05-09-rest-magpie-design.md` §3, §4, §5, §6, §10, §13, §15.
- Goes through spec-reviewer. Lock the contract before any feature PR opens.

**Phase 2.1-2.7 — feature PRs (sequential, in this order):**

1. `feat/body-mode-rename` — replace `include_body: true | false | "auto"` with `body_mode: "auto" | "schema" | "head" | "inline"`. Reject legacy `include_body` field with `unsupported_field` error containing migration hint. Other PRs in the bundle build on the new param, so this goes first.
2. `feat/body-mode-head` — implement `"head"` behavior + extend `auto` to upgrade to head between thresholds.
3. `feat/schema-next-step-hints` — auto-generate jq pattern suggestions based on top-level shape. Sibling field on response, advisory.
4. `feat/inline-cap-with-friendly-error` — `MAGPIE_INLINE_BODY_CAP` (default 256 KB) guards `body_mode: "inline"` on big bodies. New error kind `body_too_large_for_inline` with cache_id payload so agent can switch to `http_read` without refetching.
5. `feat/inclusion-metadata` — `meta.body_inclusion: { resolved_mode, inline_threshold_bytes, head_preview_threshold_bytes, reason? }` replaces `meta.body_included: boolean`.
6. `feat/server-info-tool` — new 4th MCP tool. Returns version + runtime detection + cwd + files_root + effective limits. Debug surface for path-rejection investigations.
7. `feat/multipart-content-base64` — `multipart.files.<name>` becomes `oneOf({path}, {content_base64, filename?, content_type?})`. New env `MAGPIE_MAX_INLINE_FILE_BYTES` (default 10 MB). Critical for remote-MCP scenarios.

**Gate to v0.2.0 release:**
- Spec amendment merged + all 7 feature PRs merged.
- Two-stage spec-reviewer: amendment PR + final bundle review on the last feature PR.
- pre-release-checker green.
- All test descriptions updated to use `body_mode` (housekeeping during the rename PR).
- README updated as part of the rename PR (link snippets, examples, configuration table).
- Tag `v0.2.0`, release.yml publishes.

## Phase gates and exit criteria

| Gate | What blocks moving to next phase |
|---|---|
| 0 → 1 | All Phase 0 PRs merged to main. README presentable. |
| 1 → 2 | `v0.1.2` tagged and published. No critical issues filed against it for at least 48h. |
| 2 → 3 | `v0.2.0` tagged and published. README updated with new API in the rename PR. |
| 3 → 4 | Phase 3 polish PRs merged. Copy templates reviewed by user. |

Hard rule: **promo does not start until Phase 3 ships.** Any temptation to start earlier means we burn a one-shot post on a less-polished product.

## Deferred to v0.3+

Not happening in this roadmap. Tracked here so we don't lose sight:

- Native `node-jq` engine (env-flag wired, impl skipped)
- `http_history` tool
- Proxy support
- Cookie jar
- Multi-arch GHCR builds (linux/arm64)
- Non-chunked multipart compat mode (only if a real issue is filed)
- Curl export / repro helper
- Inline base64 binary download (`download_mode: "inline_base64"`) — only if remote MCP traction surfaces a real need

Rejected outright (not even in v0.3 backlog):

- Artifact-oriented file flow (handle-based file API). Solved cheaper via `content_base64` for upload + future `download_mode: "inline_base64"` if needed.

## Two hooks — pick by audience

We have two distinct value props, and different audiences respond to different ones. Don't try to lead with both — pick the one that lands hardest for the channel.

**Hook A — Token economics (universal):**
> "Your agent burned 12K tokens on a 200KB JSON to read one field. With rest-magpie: ~200 tokens for the same field."
- Lands with anyone running an agent against a non-trivial API. Concrete numbers, immediate "I've been there."
- Default for: README, Show HN title, dev.to title, Twitter opener, all aggregator descriptions.

**Hook B — Permission fatigue (CLI-agent dev audience):**
> "Tired of approving every single `curl` your agent runs? Even after clicking 'Always allow', the next call has a slightly different URL or flag and the prompt is back. With rest-magpie, you authorize the MCP server once at config time — zero prompts after that, no matter how many HTTP calls the agent makes."
- Lands hardest with people living in Claude Code / Cursor / Cline daily. Doesn't land at all if the audience is on Claude Desktop with no bash exposure.
- Default for: Reddit r/ClaudeAI primary hook, Twitter thread tweet #3 (after Hook A in #2), dev.to "Why" section, secondary paragraph in Show HN body.
- **Caveat to keep us honest:** "zero prompts after that" is true for the MCP tool calls themselves — config-time authorization covers all subsequent invocations. Don't oversell it as "no permissions ever again," only as "no per-call permission churn for HTTP."

The two hooks reinforce each other: token economics is the *quantitative* argument, permission fatigue is the *ergonomic* argument. The combo (one MCP install solves both) is what makes the value prop unique vs alternatives like fetch-mcp (solves permissions but not tokens) or `curl + jq` in bash (solves neither, plus the prompt churn).

## What we're NOT doing

- VS Code marketplace extension wrapping. VS Code already supports MCP natively via `.vscode/mcp.json` — extension would be redundant maintenance burden.
- ProductHunt — wrong audience (SaaS-first, not dev-tools).
- LinkedIn — wrong audience for a CLI MCP server.
- Discord-spam in foreign servers — banhammer + reputation hit.
- Direct outreach to Anthropic team — they index `modelcontextprotocol/servers` automatically.
